## 三、非分离部署方案

### 3.1 构建与部署流程

```bash
# 1. 开发环境（启动前端开发服务器，后端需同时运行）
cd frontend && npm run dev        # Vite dev server (port 5173)
# 同时启动后端：cd eighteen_report && ./gradlew bootRun

# 2. 生产构建（输出到 Spring Boot static 目录）
cd frontend && npm run build

# 3. 打包 JAR 并运行
cd eighteen_report && ./gradlew bootJar
java -jar build/libs/eighteen_report-*.jar

# 4. 访问 http://localhost:9876
```

### 3.2 关键配置说明

**Vite 构建输出**（`frontend/vite.config.ts`）：
```ts
build: {
  outDir: '../eighteen_report/src/main/resources/static',
  emptyOutDir: true,   // 每次构建前清空旧文件
}
```

**开发代理**（同一配置文件）：
```ts
server: {
  proxy: { '/api': 'http://localhost:9876' }
}
```

## 四、Vue Router History 模式 + Spring Boot 404 解决方案

**问题**：用户直接访问 `/report/design/1` 或刷新页面时，Spring Boot 找不到对应路由返回 404。

**解决**：新增 `SpaController.java`，将所有前端路由 forward 到 `index.html`：

```java
@Controller
public class SpaController {
    @RequestMapping(value = {"/", "/report/**", "/datasource/**"})
    public String spa() {
        return "forward:/index.html";
    }
}
```

- 静态资源（`.js`/`.css`/`.ico`）由 Spring Boot 默认静态资源处理器优先处理，不经过此控制器
- `/api/**` 由各 API 控制器处理

## 七、报表模板数据结构

模板内容保存为 `content` 字段中的 JSON 字符串，结构如下：

```json
{
  "datasets": [
    {
      "key": "orderDs",
      "type": "SQL",
      "fields": ["id", "name", "amount"],
      "dataSourceId": 1,
      "sql": "SELECT id, name, amount FROM orders"
    }
  ],
  "cells": {
    "A1": { "value": "订单报表", "style": { "fontWeight": "bold", "fontSize": "16px" } },
    "B2": { "value": "${orderDs.name}", "type": "text" }
  },
  "merges": [
    { "row": 0, "col": 0, "rowSpan": 1, "colSpan": 3 }
  ],
  "gridMeta": {
    "rowCount": 50,
    "colCount": 10,
    "defaultRowHeight": 25,
    "defaultColWidth": 100,
    "colWidths": { "A": 120, "B": 200 },
    "freezeHeaderRows": 1,
    "watermarkCallbackUrl": "https://biz.example.com/api/watermark",
    "renderShowGridLines": false,
    "edgeMarginRightCol": false,
    "edgeMarginTopRow": false,
    "edgeMarginLeftCol": false
  }
}
```

**预览页列宽**：`ReportPreview.vue` 从同一 `content.gridMeta` 读取 `colWidths` 与 `defaultColWidth`，为 AG Grid 列设置像素 `width`，与设计器一致；总宽超过视口时出现横向滚动，不再使用 `flex` 均分整行。

**预览页运行时查询参数**：`ReportPreview.vue` 会把预览 URL 上的 query 参数读取为运行时参数，并透传给 `/api/report/render` 与 `/api/report/export`。例如访问：

```text
/report/preview/15074e28-7c06-41cf-9822-e4aa67b87d20?name=张三&department=研发部
```

则运行时参数为：

```json
{
  "name": "张三",
  "department": "研发部"
}
```

这组参数会同时影响预览、分页翻页和 Excel/PDF 导出，保证结果一致。

**页边留白（行列留白，非 CSS padding）**：设计器列/行右键菜单或报表设置配合使用——通过「插入页边留白（顶行+左右列）」在模板内插入一行顶空白行与最左、最右空白列；顶行高度与两侧列宽取自 `gridMeta.pageMargins` 的 **top / left / right**（不小于 4px）。若 `freezeHeaderRows > 0`，冻结行数会自动 +1，使原冻结表头仍对应原内容块（顶空白行进入冻结区）。保存与导出与所见一致。

**预览与留白列/行**：`ReportRenderService` 对静态行不再丢弃「输出尚未开始时的全空行」，避免顶部留白行进不了 `cells`。右侧留白列：`gridMeta.edgeMarginRightCol`（执行「插入页边留白」后为 `true`）时，预览仅在「内容+合并推断列数」基础上多渲染 **一列**，并用 `gridMeta.colCount` 做上限；同时写入 `edgeMarginTopRow` / `edgeMarginLeftCol`，预览页对该区域单元格使用 `pointer-events: none`，避免点击出现选格效果；横向滚动通过 `suppressHorizontalScroll` 按「列总宽是否大于 body 视口」动态切换，仅在内容超出时出现。

**预览网格线**：`gridMeta.renderShowGridLines`（设计器「预览/渲染显示网格线」或右键「预览渲染时显示网格线」）控制预览页 AG Grid 是否绘制主题网格线；默认 false，仅保留单元格自画边框，与设计器 `showGridLines` 独立。

### 7.2 图片上传与渲染（模板级回调）

- **定位**：报表工具本身不承担图片存储，充当"中转代理"；图片最终由业务方存储，报表工具仅保存业务方返回的 URL。

- **上传流程**：
  1. 在「报表设置 → 图片上传回调地址」中填写业务方接收接口地址，并保存到 `gridMeta.imageUploadCallbackUrl`（模板级，不同模板可接入不同业务系统）。
  2. 在 `FormatToolbar` 点击图片按钮，下拉选「本地图片」或「网络图片」：
     - **本地图片**：浏览器选文件 → 前端 POST `multipart/form-data` 到 `/api/report/{templateId}/image/upload/local`；后端读取模板的 `imageUploadCallbackUrl`，以相同格式转发（参数名 `imgFile`）到业务方回调。
     - **网络图片**：用户输入 URL → 前端 POST `{"imageUrl":"..."}` 到 `/api/report/{templateId}/image/upload/remote`；后端拉取该 URL 的图片，再转发到业务方回调。
  3. 业务方回调返回图片 URL（纯字符串或 JSON `{"url":"..."}`），写入当前锚点单元格：`type='image'`，`value=<URL>`。

- **安全**：后端对 `imageUploadCallbackUrl` 进行与水印回调相同的安全校验（协议限制 http/https、host 白名单 `WatermarkCallbackProperties.allowedHostPatterns`、私网拦截 `denyPrivateHosts`）。网络图片拉取地址也会校验协议。

- **渲染**：设计器（`ReportGrid.vue`）和预览页（`ReportPreview.vue`）通过 AG Grid `cellRenderer` 渲染 `image`（图片）、`barcode`（条形码）、`qrcode`（二维码）。
  - 预览中双击图片可弹大图预览；
- Excel 导出（`ExcelReportExporter`）会按单元格类型执行媒体嵌入：
    - `image`：下载 URL 或解析 dataURL 后嵌图；
    - `barcode` / `qrcode`：后端生成 PNG 再嵌图；
  - 媒体生成或下载失败时降级为原文本，不中断导出（保证可用性）。
  - 当前实现已重构为导出模块（`service/export/**`），Excel 由 `ExcelReportExporter` 承担，媒体解析由 `export/shared/MediaBinaryResolver` 统一处理。

### 7.3 PDF 导出

- **入口**：预览页点击「导出 PDF」，请求后端 `/api/report/export`，`format=pdf`。
- **数据流**：后端复用同一套渲染结果（变量替换 + 数据行展开后的 `cells/styles/merges`），并读取模板 `gridMeta` 的列宽/冻结行/默认行高等配置。
- **分页与冻结表头**：PDF 采用 iText7 直接绘制表格；冻结行通过 iText Table 的 header cell 机制在分页时重复显示（视觉上类似冻结表头）。
- **合并区域**：根据 `merges` 在 PDF 中为合并锚点单元格设置 `rowSpan/colSpan`，并跳过覆盖区单元格重复绘制，避免边框/内容叠加。
- **媒体绘制与降级**：`image / barcode / qrcode` 在单元格矩形内嵌入绘制；当媒体生成/下载失败时降级为原文本，保证导出可用性。
- **图片遮挡边框处理**：为避免图片贴满单元格遮挡边框线，PDF 中图片按单元格宽高的约 `75%` 进行缩放并居中绘制。
- **字体策略**：优先加载后端 `resources/fonts/` 内的中文字体并嵌入 PDF；未配置或加载失败时回退到内置标准字体（中文可能显示为方块）。
  - 当前实现已重构为 `PdfFontProvider`（优先 Noto OTF，bold 失败回退 regular，避免中文方块），水印由 `PdfWatermarkRenderer` 通过“每页事件”绘制。

### 7.4 导出模块架构（可扩展）

- **目标**：将导出逻辑从单一大类拆分为按格式扩展的组件，便于未来新增 `csv/json/image` 等导出而不改动主流程。
- **入口**：`ReportExportFacade`（统一：渲染 + 读取模板 content + format 归一化 + 路由）。
- **路由**：`ExporterRegistry`（从 Spring 容器收集所有 `ReportExporter`，按 format 选择实现）。
- **策略接口**：`ReportExporter`（每种格式一个实现，如 `ExcelReportExporter`、`PdfReportExporter`）。
- **共享能力**：`service/export/shared`（gridMeta 解析、合并索引、媒体类型解析、媒体二进制解析与 SSRF 校验），减少多格式重复代码。

### 7.5 数据集分页（设计态配置 + 预览态翻页）

- **配置位置**：设计器「数据集弹窗」按 dataset 配置分页，不再使用报表级统一分页开关。
  - `pagination.enabled`：是否对该数据集启用分页；
  - `pagination.defaultPageSize`：预览默认每页条数（默认 20）；
  - `pagination.request`：请求参数名映射（`page/pageSize/offset/limit`）；
  - `pagination.response`：API 返回字段路径映射（`recordsPath/totalPath/currentPagePath/pageSizePath/hasMorePath`，点路径）。
- **预览页交互**：在导出按钮左侧提供分页控件（页码、上一页/下一页、每页条数），并支持导出范围切换（当前页/全部）。
- **后端查询链路**：
  - `ReportQueryRequest` 增加 `page/pageSize/datasetKey`；
  - `ReportQueryResponse.DatasetResult` 增加 `pagination` 元信息（`total/currentPage/pageSize/hasMore`）；
  - `ReportRenderResponse` 增加 `paginationByDataset`，前端用于展示分页状态。
- **SQL 分页实现**：
  - 在 `DataSourceConfigService` 中新增分页执行路径（count + page query）；
  - 引入 `service/datasource/sql` 方言抽象（`SqlPaginationDialect`）与解析器（当前落地 MySQL）。
- **API 分页实现**：
  - 按 dataset 配置将页码参数注入请求（GET 追加 query，POST 合并 body JSON）；
  - 按响应路径提取数据列表与总数等分页元信息；缺失时按安全兜底返回。
- **兼容策略**：未开启分页的数据集保持原有全量查询行为，旧模板无需迁移即可继续使用。

### 7.6 动态查询参数（预览 URL -> 数据源执行）

- **入口**：预览页通过 URL query 传入，例如：

```text
/report/preview/{templateId}?name=张三
```

- **前端传递链路**：
  - `ReportPreview.vue` 从 `route.query` 读取参数并归一化为 `runtimeParams`；
  - 预览请求调用 `/api/report/render` 时写入 `params`；
  - 导出请求调用 `/api/report/export` 时写入 `queryParams`；
  - 分页控制字段 `page/pageSize/datasetKey` 与业务动态参数分离，避免把分页控制误传给业务数据源。

- **后端传递链路**：
  - `ApiReportController.render` 接收 `ReportQueryRequest.params`；
  - `ReportRenderService` 将其透传给 `ReportQueryService`；
  - `ReportQueryService` 在执行每个 dataset 时，将运行时参数传入 SQL 或 API 数据源执行层；
  - `ReportExportFacade` 导出时复用同一套 `queryParams`，确保导出结果与预览一致。

- **SQL 数据源规则**：
  - 支持在 SQL 中使用 `${paramName}` 文本占位符，例如：

```sql
select * from demo_user where name like '%${name}%'
```

  - 当 URL 为 `?name=张三` 时，执行前会先替换为：

```sql
select * from demo_user where name like '%张三%'
```

  - 当 URL 未传 `name` 时，`${name}` 会替换为空串，最终 SQL 为：

```sql
select * from demo_user where name like '%%'
```

  - 对于 `=`、`>=` 等不能自然退化为“全量”的条件，支持 **可选 SQL 条件块** 语法：

```sql
select * from demo_user
where 1 = 1
[[ and name = '${name}' ]]
[[ and age >= ${age} ]]
```

  - 规则：
    - `[[ ... ]]` 中只要引用到的任一参数缺失或为空白，该整段 SQL 会被移除；
    - 所有引用参数都存在时，保留该片段，再继续执行 `${paramName}` 文本替换；
    - 因此更适合这种“有参数才拼条件”的写法，避免未传参时变成 `name = ''`、`age >= ''`。

  - 替换完成后，若 SQL 中仍存在 `:paramName` 命名参数，则继续走现有 `PreparedStatement` 参数绑定逻辑。
  - `DataSourceConfigService` 在执行 SQL 前会打印日志：
    - 普通查询：`执行报表 SQL: ...`
    - 分页统计：`执行报表 SQL(count): ...`
    - 分页数据：`执行报表 SQL(page): ...`
  - 日志中会输出已代入参数值后的可读 SQL，便于排查数据源问题。

- **API 数据源规则**：
  - URL query 参数会自动追加到 API 数据源请求地址；
  - 若 API body 或 header 中存在 `${paramName}`，也会按同样规则做文本替换；
  - 参数值若包含中文或特殊字符，会在构造请求 URL 时自动做 UTF-8 编码。

- **水印回调兼容**：
  - `WatermarkCallbackService` 会把除 `watermark` 外的运行时参数继续透传给 `gridMeta.watermarkCallbackUrl`；
  - 透传时统一做 UTF-8 URL 编码，因此 `name=张三` 这类中文参数不会再导致回调 URI 构造失败。

- **未配置时**：点击图片上传按钮后，工具会弹出警告引导用户先配置回调地址，不执行上传。

### 7.1 动态水印（服务端回调，防前端伪造）

- **问题**：若仅靠预览页 URL 参数把 `watermark` 传给 `/api/report/render`，用户可在浏览器篡改，无保密性。
- **做法**：在 `gridMeta` 中增加可选 `watermarkCallbackUrl`（设计器「报表设置」中配置）。渲染时 **Spring 服务端** 对该 URL 发起 **GET**，Query 附带 `templateId` 及渲染请求中的其它 `params`（**不含** `watermark`，避免把客户端伪造值转给业务方）。业务返回：
  - JSON：`{"watermark":"展示文案"}`（推荐），或
  - 纯文本 body（长度受配置限制）。
- **优先级**：已配置且非空的 `watermarkCallbackUrl` 时，**忽略**请求体中的 `params.watermark`，仅以回调结果作为动态水印；回调失败、超时或非 2xx 时**仅不展示动态水印**，报表单元格数据照常返回。
- **运维**：`application.yaml` 中 `eighteen.report.watermark-callback` 可配置超时、响应体长度上限、`allowed-host-patterns`（**生产强烈建议配置**以减轻 SSRF）、`deny-private-hosts` 等。
- **其它方案**（选型参考）：业务网关签发短期 JWT，仅报表后端持有校验密钥；或水印不落 URI、由同域 Cookie/Session 在后端渲染线程内解析。当前实现为「模板存回调 URL + 报表服务出站 HTTP」，与业务常见集成方式一致。

## 八、静态资源缓存策略

Vite 构建自动为 JS/CSS 文件添加内容 hash（如 `index-B-3mzKhG.css`），Spring Boot 配置长期缓存：

```yaml
spring:
  web:
    resources:
      cache:
        period: 31536000        # 1年，适用于有 hash 的静态资源
```
